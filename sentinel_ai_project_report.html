Section 01

Project Overview

Sentinel AI is a real-time behavioral risk analysis and surveillance system designed for law enforcement, security operatives, and CCTV monitoring centers. It continuously analyzes live webcam feeds using a hybrid AI pipeline — combining computer vision, optical flow analysis, and a custom-trained violence classifier — to detect threats and alert authorities with zero manual intervention.

Mission: Transform any standard CCTV camera into an intelligent threat-detection unit that automatically identifies violent behavior, records incident evidence, GPS-tags the location, and notifies police — all within seconds of detection.

AI Detection Engine

The ML service is the core intelligence layer of Sentinel AI. It runs as a standalone Python/Flask microservice, processing live video frames through a multi-stage hybrid pipeline to detect violent behavior with high accuracy and minimal false positives.

Optical Flow (Farneback)

Measures inter-frame pixel movement magnitude. Detects rapid bursts of motion that characterize violent events. Filters out camera shake (threshold: 10.0).

👁️

YOLOv8n Person Detector

Detects people in frame with bounding boxes. Analyzes bounding-box displacement (speed), posture (aspect-ratio change), and inter-person proximity.

🥊

Fight Classifier

Custom YOLOv8n-cls model trained on SCVD dataset. Classifies fight vs. normal video content. Triggers with 82%+ confidence + minimal motion.

Temporal Violence Score (TVS)

A running score that accumulates evidence of violence across multiple frames, preventing single-frame false positives. The system requires sustained evidence over 3+ consecutive frames before any alert is raised.

Standard violence conditions+0.45 per frameMotion-based accumulation

Section 03

Violence Decision Logic

The decision engine evaluates multiple independent signals simultaneously. No single signal alone is sufficient — the system uses multi-signal corroboration to ensure high precision and avoid false alerts in police-grade deployments.

Signal Types

💨

Optical Flow Signals

  • FLOW_MICRO (0.3) — Any motion at all
  • FLOW_WATCH (2.5) — Trackable motion
  • FLOW_SIG (4.5) — Real physical activity
  • FLOW_HIGH (7.0) — Fight-level motion
  • FLOW_INSTANT (10.0) — Explosive burst
📐

Geometry Signals

  • Proximity — Persons overlapping (ratio 0.9)
  • High Energy — Displacement ≥ 45px
  • Moderate Energy — Displacement ≥ 25px
  • Posture Change — Aspect ratio Δ ≥ 0.50
  • Avg Flow — Sustained ≥ 5.0 over window
🤖

Classifier Signal

  • FIGHT_HIGH (0.82) — Strong signal, needs FLOW_MICRO
  • FIGHT_MEDIUM (0.70) — Boost, needs FLOW_WATCH
  • Used for detecting fight video on phone screen
  • Never triggers without at least minimal motion

Per-Scenario Decision Rules

👤 1 Person in Frame
  • Violent: avg_flow ≥ 5.0 AND flow_score ≥ 7.0 → "Extreme sustained movement"
  • Violent: flow_score ≥ 7.0 AND aggressive posture → "High-energy aggressive posture"
  • Violent: flow_score ≥ 4.5 AND classifier ≥ 0.70 + motion → "Fight content + motion"
  • Violent: classifier ≥ 0.82 AND FLOW_MICRO → "Fight detected on screen" (for phone/screen)
👥 2+ Persons in Frame
  • Requirement: flow_score ≥ 4.5 (FLOW_SIG) is mandatory base evidence
  • Violent: base_flow + 2 of (proximity, high/moderate energy, posture, classifier+motion)
  • Violent: base_flow + 1 signal + flow_score ≥ 7.0 → "High-energy multi-person incident"
  • Violent: classifier ≥ 0.82 + FLOW_MICRO → "Fight content detected on screen"

Police-Grade Accuracy: For 2+ persons, the system requires a minimum of 3 independent signal sources. Two people sitting, standing together, or walking side-by-side will never trigger an alert — only genuine physical confrontation with high optical flow, proximity violation, AND energy signals together will elevate the risk level.

What Does NOT Trigger an Alert

ScenarioWhy It's Safe
Two people sitting/talkingNo base flow (FLOW_SIG 4.5 not reached)
Normal walkingDisplacement below DISP_HIGH (45px threshold)
Camera bump or shakeFLOW_INSTANT_HIGH raised to 10.0

Section 04

Backend & API Layer

The backend is a Node.js/Express REST API server that handles authentication, alert management, video storage, and serves as the bridge between the frontend and ML service. Data is persisted in MongoDB Atlas (cloud) with in-memory fallback for continuity.

Security Middleware Stack

🔐

Helmet.js

Sets security HTTP headers — Content Security Policy, XSS protection, frame guard.

🌐

CORS

Strict origin whitelist including both HTTP and HTTPS localhost origins for Vite dev server.

🛡️

Rate Limiting

Login: 20 req/15min. API: 100 req/15min. Webcam proxy: unlimited (20fps stream).

API Endpoint Reference

MethodEndpointDescription
POST/api/auth/loginAuthenticate user, returns JWT token
POST/api/auth/registerRegister new user account
GET/api/alertsFetch all incident alerts (role-filtered)
GET/api/alerts/:idGet single alert with full details
PUT/api/alerts/:idUpdate alert status (Investigating/Resolved/etc)
DELETE/api/alerts/:idPermanently delete alert record
POST/api/report-incidentSubmit incident with video clip upload (Multer)
POST/api/webcam-proxyProxy base64 frame to ML service (20fps bypass)
POST/api/analyze-videoSubmit video file for full CNN analysis
GET/api/profileGet user profile data
GET/healthBackend health check + MongoDB status

Database Schema

👤

User Model

  • username, password (bcrypt hashed)
  • role: admin | police | cctv_user
  • profile: name, email, phone, organization
  • Fallback to in-memory if MongoDB offline
🚨

Alert Model

  • behavior, riskLevel, confidence, timestamp
  • videoPath (uploaded incident clip)
  • location: { lat, lon } — GPS coordinates
  • status: Pending | Investigating | Resolved
  • notes: [ { text, author, timestamp } ]

MongoDB Atlas Fallback: If the cloud database is unavailable, the backend automatically switches to in-memory storage with pre-seeded default users (admin, STOREcam, NYPD) so the system never goes completely offline.

Section 05

Frontend & User Interface

Built with React 18 + Vite and served over HTTPS (self-signed SSL via @vitejs/plugin-basic-ssl), the frontend provides a real-time surveillance dashboard. All API calls are routed through the Vite development proxy to avoid mixed-content blocking.

Key Pages & Features

🔴

Live Monitor (Webcam Feed)

  • Real-time webcam via WebRTC (getUserMedia)
  • Frame captured every 50ms (20fps) → Base64 JPEG
  • Sent to backend proxy → ML Service
  • Bounding boxes drawn on canvas overlay
  • Box color: 🔵 Normal / 🟡 Medium / 🔴 High
  • Risk badge + Analysis text displayed live
  • Violence Index, Subjects, Class, Confidence shown
  • Exponential backoff if ML service goes offline
📊

Dashboard

  • Summary stats: Total Alerts, High Risk, Pending, Resolved
  • Full alert history table with search & filters
  • Filter by: Risk Level, Status, Date Range
  • Pagination (25 / 50 / 100 items per page)
  • Embedded 5-second incident video clip player
  • GPS coordinates with Google Maps link
  • Reverse geocoding → human-readable address
  • Alert actions: Verify / Report / Investigate / Resolve / Delete
🎬

Incident Recording

  • MediaRecorder API records webcam at HIGH risk onset
  • Records for exactly 5 seconds (WebM format)
  • Uploaded automatically to /api/report-incident
  • 30-second cooldown between clips (anti-spam)
  • Manual Report button for operator-triggered reports
  • Clip stored on server, path saved in alert record
🎤

Voice Alerts (Speech Synthesis)

  • Spoken alerts via Web Speech API (SpeechSynthesis)
  • "Fighting detected — alert authorities"
  • "Assault detected — immediate response required"
  • Risk level change announcements
  • Recording start/stop confirmations
  • System online/offline status announcements
📍

GPS Location Tracking

  • Browser Geolocation API (lat/lon)
  • Attached to every incident report automatically
  • Reverse geocoded to street address (Nominatim)
  • Google Maps deep link for each incident
  • 1 req/sec rate limiting for Nominatim API
📤

Video Upload Analysis

  • Upload MP4/WebM/AVI video for offline analysis
  • Analyzed by ML service CNN + YOLO pipeline
  • Returns per-frame violence scores
  • Results stored as alert records
  • Max file size: 100MB

User Roles & Access Control

The system implements Role-Based Access Control (RBAC) with three distinct user roles, each with tailored permissions reflecting real-world security operations hierarchy.

👑 Administrator

  • Full system access
  • View all alerts
  • Update any alert status
  • Delete alerts permanently
  • Clear alert history
  • Manage user accounts
  • Configure system settings
  • Access audit logs

👮 Police Officer

  • Read-only audit view
  • View all incident alerts
  • View incident video clips
  • Mark alerts as Investigating
  • Mark alerts as Resolved
  • Delete alerts permanently
  • Add investigation notes
  • Cannot clear history

📹 CCTV Operator

  • Live monitor access
  • View own alerts
  • Mark alerts as Verified
  • Mark alerts as Reported
  • Mark as False Alarm
  • Submit manual reports
  • Clear own history
  • Upload videos for analysis

Security Architecture Notes

  • HTTPS Only: Frontend served exclusively over HTTPS (Vite basicSsl) — required by browser APIs (getUserMedia, Geolocation)
  • Proxy Architecture: All backend calls go through Vite dev proxy — avoids mixed-content blocking (HTTPS page → HTTP backend)
  • No Direct HTTP Calls: Frontend uses relative paths (/api/..., /uploads/...) — never hardcoded HTTP URLs
  • File Validation: multer enforces MIME type + extension whitelist for uploads (MP4, WebM, AVI, MKV, JPG, PNG)

Deployment & Configuration

Service Architecture

🧠

ML Service — Port 5000

Python Flask. Starts with: python -m ml_service.app. Loads YOLOv8n + fight classifier on startup (~18s warm-up). Writes logs to ml_service.log.

🗄️

Backend — Port 4005

Node.js Express. Starts with: node server.js. Connects to MongoDB Atlas on startup. Creates uploads/ directory automatically.

⚛️

Frontend — Port 5173

Vite dev server. Starts with: npm run dev. Serves on HTTPS with self-signed cert. Proxy routes /api and /uploads to backend.

Usage Guide

Live Monitoring Workflow

  1. Open https://localhost:5173 and log in with your credentials
  2. Navigate to Live Monitor from the top navigation bar
  3. Click "Connect Feed" — grant camera permission when prompted
  4. The system begins analysis immediately — bounding boxes appear around detected persons
  5. Monitor the Risk Badge (bottom-right): Low 🟢 / Medium 🟡 / High 🔴
  6. On HIGH risk: voice alert sounds, 5-second clip records automatically, incident saved to dashboard
  7. Use "Manual Report" button to manually flag a suspicious scene
  8. Click "Terminate Feed" to stop monitoring

Reviewing Incidents (Dashboard)

  1. Open Dashboard — all incidents listed with risk level, behavior, timestamp
  2. Click the location link (📍) to open Google Maps at the incident location
  3. Play the 5-second incident clip directly in the dashboard table
  4. Update status: click Verify → Report → Investigate → Resolve as investigation progresses
  5. Use search and filters to find specific incidents by behavior, risk level, or date
  6. Police role: use Investigate and Resolve buttons to manage case workflow
